# 審查清單 — 資深工程師的「壞味道」啟發式

逐維度對照。每條都是「**現象 → 為什麼是問題 → 怎麼最小修復**」的思路。
不是每條都要報;**用判斷力挑出對這次改動真正重要的**(見末尾「真問題 vs 噪音」)。

---

## 1. 正確性 / Bug

- **邊界值**:空集合、單元素、首尾元素、最大/最小、零、負數、超長輸入。新代碼在 `n=0` 和 `n=1` 下對嗎?
- **null / undefined / 空字串 / 空指標**:可能為空的值是否在使用前判過?「絕不為空」的假設有保證嗎?
- **off-by-one**:`<` vs `<=`、`len` vs `len-1`、分頁的 `offset/limit`、切片邊界。
- **錯誤處理**:異常被吞掉(空 catch)?錯誤碼沒檢查?失敗後狀態是否半完成(部分寫入)?
- **返回值與類型**:所有分支都有返回?返回 null 和返回空集合語意混淆?早返回漏了清理。
- **冪等與重試**:重複調用會重複扣款 / 重複插入嗎?重試是否安全?
- **時間 / 時區 / 浮點**:本地時區假設、夏令時、用浮點算錢、相等比較浮點。
- **狀態機**:非法狀態轉換有沒有被擋?並發下狀態是否可能跳變。
- **回歸**:這個改動有沒有悄悄改變既有調用方依賴的行為 / 默認值?

## 2. 性能(優先)

先問兩件事:**這段在熱路徑嗎?資料規模多大?** 冷路徑上的微優化是噪音。

- **算法複雜度**:循環內套循環(O(n²))?能否用 map/set 降到 O(n)?排序 / 查找選型。
- **N+1 查詢**:循環裡逐條查 DB / 發 RPC?應批量(IN 查詢 / batch / dataloader)。
- **DB**:查詢能命中索引嗎?`SELECT *` 取多餘列?缺分頁的全表掃?事務範圍過大鎖太久?
- **熱路徑分配 / 拷貝**:循環內反覆分配、字串拼接(該用 builder)、深拷貝大對象、不必要的序列化。
- **快取**:可快取的昂貴計算 / 查詢有沒有快取?快取失效與一致性想清楚了嗎?快取擊穿 / 雪崩?
- **阻塞 I/O**:在請求 / 事件循環 / 持鎖期間做同步 I/O 或網路調用?應異步 / 移出臨界區。
- **惰性 vs 預載**:是否一次性載入了用不到的大量資料?反之 N 次小請求能否合一次。
- **資源上限**:無上限的批量 / 並發 / 記憶體增長?大輸入會 OOM 嗎?需要分頁 / 流式 / 背壓。

## 3. 邊界清晰(架構與職責)

- **單一職責**:一個函數 / 類做了幾件事?>1 個改它的理由就該拆。
- **分層方向**:依賴是否單向(view → service → model)?有沒有底層反向依賴上層、業務邏輯漏進控制器 / DAO?
- **契約清晰**:函數簽名是否誠實(隱藏副作用?偷偷改入參?)?可選 / 必填、返回語意是否明確。
- **輸入校驗位置**:校驗應在**邊界**(入口 / 反序列化處)做一次,而不是散落各處或根本沒做。
- **耦合**:模組間是否通過穩定接口而非內部細節通信?改 A 必須同時改 B 是壞信號。
- **洩漏抽象**:上層是否被迫了解下層實現(SQL 片段、HTTP 細節漏到業務層)?
- **可配置 vs 硬編碼**:環境相關值(URL、超時、開關)是否硬編碼?
- **向後相容**:改了公開 API / 數據格式 / 事件結構,舊調用方 / 舊數據還能用嗎?需不需要灰度 / 版本化。

## 4. 併發 / 資源

- **競態**:共享可變狀態有沒有保護?check-then-act(先查再改)中間是否可能被插隊?
- **鎖**:鎖粒度(太粗傷吞吐、太細易出錯)?鎖順序一致避免死鎖?持鎖時做 I/O?
- **原子性**:多步操作要麼全做要麼全不做;跨資源用事務 / saga / 補償。
- **資源洩漏**:連接、文件句柄、socket、協程 / 線程 / goroutine、訂閱、定時器 — 是否在所有路徑(含異常)被釋放?用 with / defer / try-finally。
- **連接池**:是否每次新建連接而非復用池?池大小與超時合理嗎?
- **超時 / 重試 / 背壓**:外部調用有超時嗎?重試有退避與上限嗎?下游慢時上游會雪崩嗎?
- **異步陷阱**:未 await 的 promise / future?在循環裡 await(本可並發)?異常在異步邊界丟失?

## 5. 安全

- **注入**:SQL / NoSQL / 命令 / LDAP / 模板注入 — 用戶輸入是否拼進了查詢 / 命令?應參數化。
- **鑑權與授權**:這個接口 / 操作驗了身份嗎?驗了**這個用戶能否操作這個資源**(越權 / IDOR)嗎?
- **密鑰**:有沒有硬編碼密鑰 / token / 密碼?日誌、報錯、響應裡是否洩漏。
- **敏感資料落日誌**:密碼、token、身份證、手機號是否被打進日誌 / 監控。
- **輸入處理**:反序列化不可信數據(pickle / readObject)?路徑拼接導致目錄穿越?重定向 / SSRF?
- **輸出編碼**:返回給前端 / 拼進 HTML 的內容有沒有轉義(XSS)?
- **依賴**:新引入的庫是否可信、有無已知漏洞、版本是否鎖定。

## 6. 規範 / 可維護性

- **命名**:名副其實?縮寫一致?布林讀起來像判斷(`isReady` 而非 `flag`)?與倉庫既有命名風格一致。
- **與既有風格一致**:這是規範類問題的**唯一準繩** — 看周圍代碼怎麼寫,而非你的個人偏好。
- **死碼 / 重複**:注釋掉的代碼、未使用的變量 / 導入、複製粘貼的邏輯塊。
- **魔數 / 魔串**:無解釋的常量應具名。
- **註釋**:解釋「為什麼」而非「做什麼」;過時誤導的註釋比沒有更糟。
- **可測性**:純邏輯是否與 I/O 耦合到無法單測?有沒有對應測試 / 測試覆蓋了新分支與邊界?
- **可觀測性**:關鍵路徑 / 失敗分支有沒有合適的日誌 / metric / trace?日誌級別恰當嗎?
- **過度設計**:為「將來可能」加的抽象 / 配置 / 泛型 —— YAGNI。**過度抽象和重複一樣是壞味道**,按需而非預測。

## 修復建議的哲學(務必遵循)

1. **先止血,再治本**:先給能立刻安全合入的最小改動;根因若需大改,單列為「後續重構」。
2. **分層**:把建議分成「本次必修」「本次建議」「後續跟進」,讓對方能分批落地。
3. **可灰度**:涉及行為 / 數據 / 接口變更的修復,優先給能開關 / 版本化 / 雙寫過渡的路徑,而非一刀切。
4. **反對推倒重來**:除非現狀確實不可救,否則不要建議重寫整個模組;給漸進演化路徑。
5. **給得出 diff 就給**:小修復直接給 1~3 行改法,比一段描述更有用。

## 真問題 vs 噪音(如何不淪為 nitpicker)

報告前對每條自問:

- **後果是什麼?** 說不出具體後果(壞數據 / 變慢 / 出錯 / 難維護)的,降級為 Nit 或不報。
- **在熱路徑 / 真實規模下成立嗎?** 冷路徑的微優化、不可能達到的規模 —— 不報或標 Nit。
- **是真問題還是我的偏好?** 與倉庫既有風格一致的,即使非我所好,**不報**。
- **能 actionable 嗎?** 給不出修復方向的「感覺不太好」,不報。

**寧可少報幾條高信號,也不要用一堆 Nit 淹沒真正的 Blocker。** 一份好 review 讓人一眼看到最該修的三件事。
